# Introduction

> **本章の位置付け：**
>
> 本章では、Project Sprint の全体像と、本文書群全体との向き合い方を示す。<br>
> Project Sprint は、個別の手法を理解することで成立するものではなく、体験と前提、構造を往復しながら理解されるものである。<br>
> そのため、本章では、どのような順序で各要素に触れるべきかを示し、理解と実践の導線を提示する。<br>
> Project Sprint 未体験の読者は、[Workshop](https://gemini.google.com/gem/1sSD-IgnFHhASaWH52fsn3dHWkyMg9J9L?usp=sharing) を体験したあとに本文書群を再読することで、より深い理解が得られる。
>

>
> - Workshop・CODE・Practical Guide というProject Sprint の3つの要素の関係を示し、どのような順序で理解と実践を進めていくべきかの道筋を提示する。
> - Project Sprint の背後にある思想や価値観、目指す世界のあり方を示す。
>
> **読者に提供する価値：**
>
> - Project Sprint の全体像と向き合い方を理解することで、今何を読み、どのような実践を行うべきかを自ら判断できるようになる。
> - 自身の経験を、Project Sprint 的価値観に照らして意味づけし、価値を把握できるようになる。


## 1. Project Sprint が定義する「プロジェクト」とは何か

本文書群において、「プロジェクト」という言葉は、従来とはやや異なる前提をもとに用いられる。

一般にプロジェクトは、あらかじめ定めた目標や正解に向かって計画を実行する活動として捉えられることが多い。しかし、実際のプロジェクトは不確実な状況の中にあり、環境や要求は変化しつづける。そのため、最初に定めた前提や計画は、そのまま実行することがしばしば難しくなる。

Project Sprint におけるプロジェクトとは、変化しつづける状況の中で、どのように活動を収束させるかをメンバーの意志によって決定し、その結果をもとに次の方向を見出していく活動である。

プロジェクトは、あらかじめ与えられるものではなく、意図的に切り出されるものである。そして、そのゴールは固定されたものではなく、行動と対話を通じて形成されるチームの意志によって更新されていく。

固定的な正解に向かって進むものではなく、状況に応じて価値と方向性を見出していく活動をプロジェクトとして、本文書群をお読みいただきたい。


## 2. Project Sprint は何を変えようとしているのか

世界は不確実な状況の中にある。それはわたしたちが向き合う個々のプロジェクトも同じである。環境や要求は変化し続け、将来を完全に予測することはできない。

従来のプロジェクト観では、この不確実性に対して「より正確な計画」を立てることで対処しようとしてきた。あらかじめ正解を定義し、それを効率的に実行することがプロジェクトの成功であると捉えられてきた。<br>
この前提に立つと、計画は常に現実に先行し、行動はその実行として位置づけられる。しかし現実には、計画の前提は容易に崩れ、プロジェクトはそのたびに修正を強いられる。やがて計画の維持そのものが目的化し、プロジェクトは前進の実感を失っていく。

問題は計画の精度ではなく、プロジェクトの捉え方そのものにあるのではないだろうか。<br>
Project Sprint は、このプロジェクト観を転換する。

Project Sprint におけるプロジェクトは、あらかじめ定めた正解を実行する活動ではない。不確実な状況の中で、行動を通じて理解を更新し、価値を見出していく活動である。
言い換えれば、プロジェクトとは「探索」である。探索においては、計画は出発点に過ぎず、前進は行動によってもたらされる。行動とその結果の出力が新たな理解を生み、次の意思決定を可能にする。

このとき重要になるのが、プロジェクトに関わる「個」の存在である。人はそれぞれ異なる経験や知識、関心を持っており、世界の理解は常に個人的で部分的なものである。 一人の人間が捉えられる現実には限界がある。不確実な状況においては、この限界がそのままプロジェクトの限界になる。したがって、プロジェクトがよりよい方向へ進むためには、単一の視点ではなく、多様な視点を取り込む必要がある。

Project Sprint は、個々のメンバーが持つ異なる認識や感覚を、行動と出力を通じてプロジェクトに持ち込み、それを対話によって統合していくことを前提としている。多様な個がそれぞれの視点から探索に参加することで、プロジェクトは単一の理解に閉じることなく、より豊かな価値の実現へと向かっていく。

このとき、メンバーは単に役割を遂行するだけの存在ではない。それぞれが自らの関心や違和感、気づきをもとに行動し、その結果をプロジェクトに持ち込む主体である。Project Sprint は、こうした個々の主体的な行動を前提とする。 メンバーが自らの能力を発揮し、自発的に探索に参加することではじめて、プロジェクト全体としての探索が成立する。

Project Sprint は、こうした前提に立ち、プロジェクトの進め方を再構成する。それは、あらかじめ定めた価値を実現するプロジェクトから、価値を探索しつづけるプロジェクトへの転換である。プロジェクトの目的は、最初に定めた問いに答えることではない。行動と出力を通じて問いそのものを更新しながら、その時点でよりよいと感じられる価値の実現へ向かうことである。そのため、プロジェクトにおいて重要なのは、計画への適合ではなく、変化する状況の中で問いと目的を更新しつづけることである。

Project Sprint は、プロジェクトをうまく進めるための方法ではない。プロジェクトの捉え方を変え、人とチームが価値を立ち上げる過程そのものを扱うフレームワークである。


## 3. Project Sprint が提供する価値

Project Sprint は、不確実な状況におけるプロジェクトを前進させるために、 「探索」と「収束」という2つの活動のリズムを設計する。<br>
探索とは、行動を通じて状況を理解し、可能性を広げていく活動である。<br>
収束とは、その結果をもとに次の行動の方向性を定める活動である。<br>
プロジェクトは、この探索と収束の連続によって、徐々に焦点を定めながら前進していく。

Project Sprint が提供する価値は、この探索と収束が連続する構造を通じて、プロジェクトが前進しつづける状態と、自律的な協働が成立する環境を実現することである。Project Sprint を理解し実践することで、プロジェクトの進み方そのものが変化する。

まず、プロジェクトは計画ではなく出力によって前進するようになる。<br>
個人の行動の結果として生み出される作成物や気づきが、チームが世界を理解するための一次情報となり、次の意思決定の基盤となる。プロジェクトは、事前に定めた計画をなぞるのではなく、探索による出力を起点として定期的な収束により理解を更新しながら進んでいく。
このとき、行動を通じて得られた情報が、次の意思決定を方向づける。あらかじめ正解を定めるのではなく、実際の行動とその結果をもとに、この先の方向性が徐々に明らかになっていく。

また、プロジェクトへの関わり方も変化する。<br>
Project Sprint においては、多様なメンバーがそれぞれの視点から自律的に探索に参加することが前提となる。人はそれぞれ異なる経験や関心を持ち、異なるかたちで世界を捉えている。不確実な状況においては、この多様性こそがプロジェクトの理解を広げる源泉となる。
単一の視点では捉えきれない現実に対して、複数の視点が持ち込まれることで、プロジェクトはチームにとってよりよいと納得できる方向へ進んでいく。現代のプロジェクトにおいては、多様であることは選択肢ではなく前提であり、その多様性を活かせるかどうかが成果を左右する。
Project Sprint は、個々のメンバーが自らの視点に基づいて行動し、その結果をプロジェクトに持ち込むことを可能にする。これにより、プロジェクトは特定の個人や役割に依存することなく、チーム全体として価値の実現に向かって進むことが可能になる。

さらに、意思決定のあり方も変化する。<br>
Project Sprint においては、意思決定は「正しさ」ではなく「納得」に基づいて行われる。唯一の正解を求めるのではなく、チームが次の行動を引き受けられる状態を重視する。
納得に基づく意思決定は、メンバーの主体的な行動を引き出す。自らの判断に納得しているとき、人はその行動を自分のものとして引き受けることができる。その結果、管理によって動かされるのではなく、自律的に動くチームが生まれる。

Project Sprint は、こうした変化を通じて、プロジェクトを「計画を実行するもの」から、「探索を通じて価値を見出しつづけるもの」へと転換する。


## 4. Project Sprint をどのように体験・理解・実践するか

Project Sprint の理解は、読むことだけでは成立しない。その価値は、行動と体験を通じてはじめて認識される。<br>
そのため、Project Sprint では、可能な限り以下の順序で理解と実践を進めることを推奨している。

- Workshop：探索と収束のリズムを体験する
- CODE：体験を言語化し、構造として理解する
- Practical Guide：実際のプロジェクトに適用する

Workshop では、プロジェクトにおける探索と収束のサイクルを実際に体験する。<br>
この体験は、必ずしも明確な言葉として理解されるものではないが、行動と結果を通じた手応えとして残る。

CODE は、その体験に意味を与えるためのものである。<br>
Workshop で起きていたことを言語化し、Project Sprint の前提や構造として捉え直すことで、体験を再解釈できるようになる。

Practical Guide は、その理解を実践へと接続するためのものである。<br>
実際のプロジェクトにおいて、どのように行動し、どのように進めていくかを具体的に示す。

この順序は、単なる学習の効率ではなく、Project Sprint の前提そのものに基づいている。人は、あらかじめ意味を完全に理解してから行動することはできない。行動とその結果を経験し、それをあとから解釈することで、はじめて理解が成立する。<br>
したがって、まず体験があり、そのあとに理解がある。Project Sprint は、この順序に従って設計されている。<br>
この流れを通じて、Project Sprint は知識としてではなく、実際に機能する枠組みとして理解される。


## 5. CODE の設計思想と構造

Project Sprint は、手順としてそのまま導入するものではなく、その背景にある考え方を理解したうえで受け入れることで、はじめて効果を発揮する。<br>
Project Sprint は汎用的に適用可能であるが、その効果は背景にある前提や考え方をどの程度共有できるかに依存する。<br>
これらの前提に立たずに構造だけを導入した場合、プロジェクトの進め方との間にズレが生じ、その効果は十分に発揮されない可能性がある。

CODE は、上記の前提に基づき、Project Sprint の考え方と構造を言語化したものである。<br>
ただし、それはあらかじめ理解してから適用するための理論ではない。Workshop における体験を前提として、それに意味を与え、何が起きていたのかを構造として理解できるようにするとともに、その背景にある前提や考え方を提示し、実践における判断の基準を与えることが、本文書の役割である。

したがってCODEにおいては、Project Sprint の理論や構造だけでなく、その前提となる考え方も含めて提示する。

なお、CODEは、理解の進度と、その先の実践の難度に応じて、3つのステップに分かれている。

[Step0](step0.md) では、個人が主体的に行動し、出力を通じて世界の理解を更新することを扱う。<br>
[Step1](step1.md) では、複数の個人が出力と対話を通じて共通了解を形成し、チームとして探索を進めることを扱う。<br>
Step2 では、その探索の結果として、チームがどのように価値を生み出し、社会に変化をもたらすのかを扱う。<br>
（Step2 は、v5.1以降で記述されるため、v5.0時点では記述されていない。）

## 6. CODE 読後の読者への期待

この文書群を読み終えたとき、読者はProject Sprint を完全に理解している状態になるわけではない。また、ここに書かれている内容をそのまま適用できる状態になることを目指しているわけでもない。

本文書群を通じて得られるのは、プロジェクトにおいて何が起きているのかを捉えるための視点である。<br>
Workshop での体験や、実際のプロジェクトにおける出来事に対して、それらがどのような構造のもとで起きているのかを言語化し理解できるようになる。

また、本文書群は、実践における判断の基準を与える。プロジェクトの中で何を優先すべきか。どのように意思決定すべきか。どのようにメンバーと関わるべきか。これらに対して、唯一の正解を与えるものではないが、Project Sprint の前提に基づいて考えるための軸を持つことができる。

その結果、読者は、Project Sprint を単なる手法としてではなく、状況に応じて解釈し、選択し、適用できる枠組みとして扱えるようになる。Project Sprint は、理解してから使うものではない。使いながら理解が深まっていくものである。本文書群は、その理解が成立するための足場を提供する。


ではまずは、Project Sprint におけるもっとも基本的な単位、「個人」に着目した [Step0](step0.md) から読み始めよう。
